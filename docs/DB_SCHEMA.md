## 1. Design Principles

- **Minimal and purposeful** — only tables needed for Phase 1 are defined
- **Django ORM managed** — all DDL is generated by Django migrations; no manual SQL schema management
- **Phase 2 extensible** — nullable foreign key columns and app separation allow new tables to be added without modifying existing ones
- **Soft enables/disables** — content rows use `is_active` boolean rather than hard deletes to preserve data
- **Audit timestamps** — every table has `created_at` and `updated_at`

## 3. Naming Conventions

| Rule | Convention |
|---|---|
| Table name | Django default: `{app_label}_{modelname}` in lowercase (e.g., `portfolio_service`) |
| Column names | Lowercase snake_case |
| Primary key | `id` — INT UNSIGNED AUTO_INCREMENT (Django default) |
| Foreign keys | `{model_name}_id` (Django default) |
| Timestamps | `created_at`, `updated_at` — DATETIME auto-managed |
| Boolean flags | `is_{adjective}` (e.g., `is_active`, `is_featured`) |
| Status fields | VARCHAR with defined allowed values (validated at model/serializer level) |
| Slugs | `slug` — VARCHAR(255) UNIQUE — URL-safe identifier auto-generated from name |

**Note on Django naming:** Django automatically creates the table `{app_label}_{model_class_name_lowercase}`. The model class names below correspond to table names as: `Service` → `portfolio_service`, `ContactRequest` → `leads_contactrequest`, etc.

## 7. Relationship Diagram

```
portfolio_category
       │
       │ 1
       │
       ▼ M
portfolio_item
  (slug, title, thumbnail, description, is_featured, sort_order)


portfolio_service          portfolio_product
  (standalone)               (standalone)


leads_contactrequest       leads_consultationrequest       leads_demorequest
  (standalone)               (standalone)                   (standalone)


auth_user  ←──────  django_admin_log  ──────►  django_content_type
  (Django)              (audit trail)              (any table)
```

**Phase 1 relationships are intentionally minimal:**
- `portfolio_item` → `portfolio_category` (many-to-one) is the only foreign key between custom tables
- All lead tables are standalone
- No foreign keys from lead tables to service/product tables (service/product names stored as plain text for resilience — deleting a service won't orphan leads)

## 8. Indexes

Django creates indexes automatically for:
- All primary keys
- All `ForeignKey` columns
- All `SlugField` columns (unique index)
- All `UniqueField` combinations

**Additional manual indexes (add in Django migration):**

```sql
-- Fast status filtering for admin lead views
CREATE INDEX idx_contact_status ON leads_contactrequest (status, created_at DESC);
CREATE INDEX idx_consultation_status ON leads_consultationrequest (status, created_at DESC);
CREATE INDEX idx_demo_status ON leads_demorequest (status, created_at DESC);

-- Fast public content queries
CREATE INDEX idx_service_active_order ON portfolio_service (is_active, sort_order);
CREATE INDEX idx_product_active_featured ON portfolio_product (is_active, is_featured, sort_order);
CREATE INDEX idx_portfolio_active_category ON portfolio_item (is_active, category_id, sort_order);
```

## 9. Phase 2 Extension Points

The Phase 1 schema is designed so Phase 2 tables can be added as new Django apps without altering existing tables.

| Future Feature | New Table(s) | Link to Existing |
|---|---|---|
| Blog | `blog_post`, `blog_category`, `blog_tag`, `blog_post_tags` | Standalone new app |
| Service detail pages | `portfolio_service` already has `description` field | Add `meta_title`, `meta_description` columns via migration |
| Product detail pages | `portfolio_product` already has `description` field | Add `meta_*` columns |
| Portfolio detail pages | Add `portfolio_item_image` (M2M gallery) | FK → `portfolio_item.id` |
| Career portal | `careers_jobposting`, `careers_application` | Standalone new app |
| Newsletter | `marketing_subscriber` | Standalone |
| Technologies normalised | `core_technology`, `portfolio_item_technologies` (M2M) | Replaces `technologies` text field |
| Admin notes history | `leads_statuschange` | FK → each lead model |
| Team members (editable) | `company_teammember` | Standalone |
| Partners (editable) | `company_partner` | Standalone |
| Site settings | `core_sitesettings` | Singleton model |
